The reality today, and what we need

The builders page promises "drop a repo → it runs the same afternoon." Today that happens by hand, not by button. This page keeps us honest about which parts are real, which are concierge, and which are still to build — so nobody reads the marketing as a spec.

What the page promises

Bring an AI-built repo; it's running on our infrastructure the same afternoon — compute, AI, memory, and storage provisioned, a live URL, you own the company. That's the destination. It's the right north star. It is not yet the mechanism.

What we need to build

To turn concierge into the button the page describes, in rough dependency order:

  1. Repo→run onboarding pipeline — containerize and deploy an arbitrary simple stack to Cloud Run automatically. This is what turns "we'll get it running by hand" into a button.
  2. The slop filter at intake — every repo that wants to be a vertical passes cascading gates before it runs: mechanical (build, license, no malware) → Notary risk-classify (PROMPT_ONLY / TOOL_USING / EXECUTABLE; egress surface) → intent check (does the code do what the README claims) → sandbox run (crash-test, phone-home behavior) → Governor/Canon (output meets the creator's taste). This is Notary's job, extended from skills to verticals. AI authorship is fine — 90%+ of repos will be AI-generated; what we filter is slop, not AI.
  3. Managed inference in prod — the LLM gateway consumable by live verticals, not just in principle. Sensarra is the first place to wire it.
  4. Safe-execution / sandbox layer — run third-party code without risking the platform. This is Notary's job; it isn't wired for arbitrary apps.
  5. Engine external API surface blocks self-serve · Pavel's cadenceThe control plane the automated path calls. The self-serve version can't ship without it.
  6. Per-vertical usage metering — meter each business against the shared credit. The honest, non-counsel-gated half of the economics.
  7. Settlement / payout counsel-gatedCross-vertical value and earning. Needs a regulated partner. Latest, not soonest.

Honest sizing: this is a multi-month build, and the automated path is partly blocked on Pavel's engine API. The concierge path needs none of it — which is exactly why we lead with concierge.

What stops the slop-farm

The fair stress-test: what happens when AI sends you 1000 repos at once — do you become a slop-farm, or do you filter?

We filter. Three structural reasons:

The moat is the seity, not the typing. AI will write most of what lands here, and that's fine. What we refuse is the model where an agent generates slop at scale and a human collects passive rent — that's a different company's product, and it's exactly the thing we're not. A founder using Claude to build 50 products and steering each one is a real founder; the filter targets slop, not AI authorship.

The intake gates are Notary's job. The same certification gauntlet that vets a skill vets a vertical — risk-classified, sandboxed, intent-checked. "We run real businesses, so what runs is vetted."

The economics finish the job. SEIT is closed-loop and non-redeemable: every sandbox run, every intent check costs the founder real credits they paid $1 each for. No rate limit (a real founder iterating fast should be able to) — but slop costs money before it ever reaches a customer, and the ledger does the work without anyone moralizing about it.

The one rule

The builders page is marketing, not a spec.

"Ship this" means the page + a concierge apply flow + hand-onboarding the first founders. It does not mean "go build the drop-any-repo runtime." That runtime is the roadmap above — building it off the back of the marketing page burns months and double-books against Pavel's cutover.

If you're an engineer reading the builders page: build the page and the form, not the machine it describes.
